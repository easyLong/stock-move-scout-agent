# 股票异动侦察评价体系

目标：每次改系统后，都能判断它是不是更好用了，而不是只看“这次跑通没跑通”。

当前先做两层评价：

1. 链路详情评价：这条链路解决了什么，还有什么隐患。
2. 耗时效率评价：跑一轮要多久，慢在哪里，是否值得继续优化。

## 第一层：链路详情评价

这一层回答：系统现在能不能稳定地产出“为什么涨”的可用证据。

### 评价维度

| 维度 | 要看什么 | 当前判断口径 |
| --- | --- | --- |
| 入口有效性 | 涨速榜是否能及时发现异动股票 | 通达信行情服务器每 60 秒计算全市场涨速 Top10 |
| 证据覆盖 | Top10 股票是否都能拿到社区热帖 | 成功股票数 / Top10，总热帖数，每只股票热帖数 |
| 正文完整性 | 热帖正文是不是完整，而不是截断文本 | `post_id` 详情缓存 + `/statuses/show.json` 补完整正文 |
| 解释相关性 | 热帖是否真正围绕该股票上涨原因 | 主题词、样本文本、是否出现业绩/订单/芯片/AI/资金/公告等线索 |
| 风控稳定性 | 是否频繁触发滑块、登录、WAF | 记录 `need_manual_verify`、失败股票、失败发生阶段 |
| 输出可读性 | 用户能不能快速查看和判断 | HTML/MD 热帖报告 + 证据 CSV + 一句话总结 |
| 数据保护 | 失败时是否覆盖上一轮好结果 | 整轮无帖子时不覆盖 `latest` 文件 |

### 当前链路解决了什么

| 环节 | 已解决的问题 |
| --- | --- |
| 通达信行情 | 避开网页涨速榜不稳定问题，直接用行情服务器协议获取全市场数据 |
| 涨速榜计算 | 每 60 秒自动产出 Top10，形成股票异动入口 |
| 雪球采集 | 不再直接 Python 请求雪球，而是使用已登录 Chrome 页面内 fetch，降低 WAF 拦截概率 |
| 详情正文 | 通过 `post_id` 详情缓存保存完整帖子，复跑时不重复请求详情接口 |
| 反验证处理 | 触发滑块时暂停人工验证，不做自动绕过 |
| 结果查看 | 输出 HTML/MD/CSV，方便人工查看和模型总结 |
| 失败保护 | 如果整轮失败且没有帖子，不覆盖上一轮有效结果 |

### 当前主要隐患

| 隐患 | 影响 | 后续优化方向 |
| --- | --- | --- |
| 雪球滑块验证 | 某些股票会中断采集，导致该股票无社区证据 | 降低详情请求频率、增加缓存命中、人工验证恢复机制 |
| 热帖不等于真原因 | 社区观点可能是情绪、猜测或误导 | 引入公告、新闻、板块联动、资金、盘口持续性做交叉验证 |
| 长文质量参差 | 有些大 V 长文看似完整，但事实可能错误 | 模型总结时增加“证据强弱”和“待核验点” |
| 只抓雪球单源 | 社区证据覆盖有限 | 后续加入东方财富股吧、公告、新闻、研报、互动易等证据源 |
| 盘后复盘和盘中实时混在一起 | 时效判断不够细 | 增加 `captured_at` 与帖子时间的时差评价 |
| 模型总结依赖输入质量 | 输入垃圾会导致总结垃圾 | 先做证据结构化，再做模型判断 |

## 第二层：耗时效率评价

这一层回答：系统是不是足够快，慢在哪里，优化是否有效。

### 评价维度

| 维度 | 要看什么 | 判断口径 |
| --- | --- | --- |
| 总耗时 | 从开始到报告产出用了多久 | `end_time - start_time` |
| 单股耗时 | 每只股票采集平均多久 | 总耗时 / 成功股票数 |
| 行情耗时 | 全市场行情和涨速榜计算多久 | 通达信模块耗时 |
| 雪球列表耗时 | 打开股票页 + 列表 fetch 多久 | 每只股票列表阶段耗时 |
| 详情补抓耗时 | 缺失 `post_id` 详情补抓多久 | `detail_fetch_count * 单次详情耗时` |
| 缓存收益 | 缓存命中节省了多少详情请求 | `cache_hit / (cache_hit + detail_fetch)` |
| 人工等待 | 是否等待滑块验证 | `manual_verify_wait` 实际等待秒数 |
| 输出生成耗时 | HTML/MD/模型总结耗时 | 报告生成和模型调用耗时 |

### 当前耗时基准

从最近测试看：

| 测试 | 结果 |
| --- | --- |
| 单只股票 3 条热帖，首次抓详情 | 约 46 秒，`cache_hit=0; detail_fetch=3` |
| 同一股票复跑 3 条热帖，全部命中缓存 | 约 29 秒，`cache_hit=3; detail_fetch=0` |
| 14:06:02 这批 Top10，每只 3 条热帖 | 约 530 秒，9 只成功，1 只触发滑块 |

### 当前效率判断

| 环节 | 当前效率 | 判断 |
| --- | --- | --- |
| 通达信涨速榜 | 快 | 适合作为实时入口 |
| 雪球列表 fetch | 中等 | 仍依赖 Chrome 页面操作和 snapshot |
| 详情补抓 | 慢且容易触发风控 | 已通过 `post_id` 缓存降低重复请求 |
| HTML/MD 报告 | 快 | 可以保留 |
| 模型总结 | 中等到慢 | 适合放在证据采集之后，不应阻塞行情刷新 |
| 整体 Top10 社区采集 | 偏慢 | 适合“异动后证据补充”，暂不适合每分钟全量深抓 |

## 当前阶段结论

当前系统已经适合作为“涨速异动后的社区证据采集器”，但还不适合作为“每分钟对 Top10 全量深度采集”的实时系统。

更合理的当前使用方式：

1. 通达信模块每 60 秒持续跑，负责发现异动。
2. 雪球模块只对重点股票或触发条件股票做证据补充。
3. `post_id` 缓存持续积累，降低复跑成本。
4. 模型总结只消费结构化证据，不参与前置抓取。

下一步最该优化：

1. 在运行日志里自动记录每个阶段耗时。
2. 在输出报告里显示缓存命中率、失败原因和本轮链路评价。
3. 把“为什么涨”的证据分为社区、公告、新闻、板块、资金五类，逐步做交叉验证。
