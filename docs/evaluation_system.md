# 验证与评审清单

本文用于快速检查当前链路是否可信，不再沿用旧的评分体系文档。

## 验证目标

每次重跑数据或修改链路后，重点确认三件事：

1. 研究池是否正确。
2. 有效事实是否干净。
3. 页面展示是否快且不展示旧证据。

## 研究池检查

检查点：

- 情绪票是否来自近 5 日涨停池。
- 趋势票是否来自近 5 日无涨停且 5 日涨幅 Top30。
- 是否剔除 ST 和北交所。
- `source_kind` 是否只有当前规则需要的值。

示例 SQL：

```sql
SELECT source_kind, COUNT(*)
FROM research_pool_items
WHERE trade_date = '2026-05-15'
GROUP BY source_kind;
```

异常信号：

- 趋势票数量不是 Top30 附近。
- 出现 ST 或北交所。
- 研究池只有 50 条，通常说明仍在读旧问财 Top50 口径。

## 有效事实检查

检查点：

- 是否只来自 `stock_ths_root_items`。
- 是否只展示近 10 日 F10 重要事件。
- 是否包含事件详情中的关键数字。
- 普通龙虎榜是否被排除。
- 知名游资/机构明显买入类龙虎榜是否被保留。

示例 SQL：

```sql
SELECT source_table, evidence_group, evidence_role, COUNT(*)
FROM stock_effective_facts
WHERE trade_date = '2026-05-15'
GROUP BY source_table, evidence_group, evidence_role;
```

异常信号：

- 出现 `ths_stock_concept_explanations` 作为有效事实。
- 出现大量普通融资融券、股东人数、分红、大宗交易。
- 页面硬催化里出现 10 天以前的旧消息。

## 模型总结检查

检查点：

- 没有有效事实的股票不应调用模型。
- 有效事实 hash 不变时应复用总结。
- 总结里不能丢掉日期、金额、比例、股权、订单、客户等关键指标。
- fallback 总结可读，但应标记来源。

示例 SQL：

```sql
SELECT code, source_generation, updated_at
FROM async_evidence_summaries
WHERE trade_date = '2026-05-15'
ORDER BY updated_at DESC
LIMIT 20;
```

异常信号：

- 大量无有效事实股票也有模型总结。
- 总结只有空泛判断，没有事实数字。
- 同一批事实反复触发模型。

## 异动情报流检查

检查点：

- 只研究当前研究池范围内股票。
- 实时领涨和稳定异动区分清楚。
- 有效事实总结能在列表和详情中看到。
- 题材标签不带旧前缀。

异常信号：

- 14:23 前的数据消失。
- 明显异动股票没有进入流。
- 涨速、成交额变化和窗口强度不一致。

## 市场概览检查

检查点：

- 非交易日显示最近一个交易日数据。
- 折线只显示交易时间。
- 研究池样本数应为当日全部研究池，不是固定 50。
- 成交金额是全市场成交金额，不是成交量。

异常信号：

- 研究池折线样本固定 50。
- 历史缺失导致五日结构断档。
- `research_pool_*` 为空或研究池样本数异常。

## 领头羊检查

检查点：

- 全市场组和题材组都基于当前研究池/题材规则。
- 情绪票 Top 和趋势票 Top 展示边界清楚。
- 公司亮点、有效事实总结可读。
- 个股题材只在主页题材/当前头条题材维度中选。

异常信号：

- 旧的“近3池”等标签还出现。
- 分数相同排序不稳定。
- 亮点或总结显示乱码、问号。

## 页面性能检查

目标：

- 异动情报流首屏主要读盘中表和缓存。
- 证据详情优先读 `stock_root_evidence_cache`。
- 复杂有效事实和模型总结不应在打开页面时实时运行。

异常信号：

- 切换题材卡片明显卡顿。
- 打开详情时触发大量 SQL。
- 页面请求等待模型总结。
